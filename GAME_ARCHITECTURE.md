## Current Working Shape

The current concept is a large-scale 3D world map viewer and exploration foundation built in Godot with GDScript. The world is procedurally generated by a separate C# terrain creation tool, saved as a `world.json` manifest plus per-chunk binary height data, and then treated as a static editable world for runtime viewing and later gameplay. Height data is stored as 16-bit unsigned integer samples. Godot imports the generated data and builds terrain meshes for viewing at a render scale of `1 Godot unit = 10 meters`. The full target world is about 1000 km x 1000 km, divided into 10 km x 10 km chunks. Each prototype chunk uses 129 x 129 height samples. The player starts by navigating the world itself with a free-orbit camera: panning, scrolling, rotating, and zooming from a broad world-map view into local terrain detail.

This should be treated as the foundation for a game, not merely a decorative map. Terrain scale, navigation, asset placement, settlement markers, zones, and future simulation hooks should be considered early enough that the map can later support gameplay.

## Early Architecture Implications

- The world should be divided into chunks or tiles so only nearby/high-detail areas need full rendering.
- Terrain needs multiple levels of detail so zoomed-out navigation remains fast while zoomed-in views show richer local detail.
- The camera system is a primary feature, not a debug tool. It needs smooth pan, zoom, and likely rotation controls.
- The static world data should be stored separately from runtime rendering code.
- World authoring needs a path: either external tools, generated source assets, an in-game/editor tool, or some combination.
- The renderer should support sparse detail: landmarks, roads, biome boundaries, ruins, settlements, resources, and other points of interest can appear based on zoom level.
- World scale must be separated from render scale. A 1000 km world should map to manageable Godot coordinates through chunk origins, floating origin techniques, or coordinate rebasing.
- The procedural generator is a separate C# content pipeline tool. It should generate terrain data/assets that Godot can load without requiring C# in the game runtime.
- GDScript owns gameplay, camera logic, scene orchestration, streaming, and Godot-facing editor/runtime behavior.
- The C# terrain tool owns terrain generation algorithms, import/export, bulk data processing, and reproducible world builds.
- The file format between the C# tool and Godot is a major boundary. It should be explicit, versioned, and simple enough for GDScript to parse or import reliably.
- The first interchange format should favor engine independence over editor convenience: raw terrain chunks and metadata first, Godot-specific generated resources later if needed.
- Godot mesh generation should be deterministic from the exported chunk data, so generated meshes can be cached or rebuilt without changing the authored terrain.
- Heightmap chunks should include enough metadata for positioning, scale, resolution, neighbor relationships, and future biome/material assignment.
- `world.json` should be human-readable and versioned. It should contain global world scale, generator info, chunk grid information, and paths to chunk files.
- Per-chunk height files should be optimized for bulk loading and compact storage, while the manifest remains the readable coordination layer.
- The first prototype should be able to load a small subset of chunks from the same format intended for the full 1000 km world.
- Height samples are normalized unsigned 16-bit values from 0 to 65535.
- The manifest must define how normalized height samples map to world elevation, using global `min_height` and `max_height` values.
- The basic decode rule is: `height = min_height + sample / 65535.0 * (max_height - min_height)`.
- Avoid per-chunk elevation ranges in the first prototype because they complicate borders, editing, and visual consistency.
- Chunk binary files should not include custom headers for the first prototype.
- Godot validates chunk binary files by comparing the file byte length against the dimensions and 16-bit sample size declared in `world.json`.
- `world.json` should record scale explicitly, including meters per Godot unit, so the C# terrain tool can use real-world meters while Godot renders in reduced coordinates.
- A 1000 km world maps to roughly 100,000 Godot units across at this scale, so the runtime still needs chunk-local origins, streaming, and likely coordinate rebasing/floating origin behavior as detail increases.
- Each 10 km chunk maps to 1000 Godot units across at the initial render scale.
- The full world chunk grid is 100 chunks by 100 chunks, for 10,000 chunks total.
- Prototype generation should support smaller rectangular chunk ranges so the viewer can be tested without creating the whole world.
- A 129 x 129 sample grid gives 128 intervals across a 10 km chunk, or about 78.125 meters per interval.
- At 16 bits per sample, each raw chunk height file is 129 * 129 * 2 = 33,282 bytes before any filesystem overhead or compression.
- This prototype resolution is intentionally coarse; it is meant to prove generation, import, chunk loading, camera navigation, and mesh construction before adding close-up detail systems.

## Repository Layout

- `godot/`: Godot project and GDScript runtime/viewer code.
- `terrain_tool/`: separate C# terrain generation/export tool.
- `world_data/`: generated `world.json` manifests and per-chunk binary height files.

## Open Questions

- Is the first milestone only a 3D world viewer, or should it include basic player units/squads like Kenshi?
- What visual style should the world use: realistic, stylized, low-poly, painterly, or another direction?
- Do we need weather, day/night, fog, or atmosphere in the first prototype?
- Should settlements, roads, faction territories, and biome regions be part of the first static world data model?
- Should implementation prioritize a quick playable prototype or a more durable engine architecture?
- What Godot version should we target?
- Should hand edits happen inside Godot, inside a separate editor, or by modifying exported terrain files?

## Ideas Under Consideration

- Build a first prototype as a navigable terrain viewer before adding characters, factions, combat, or economy.
- Use chunked heightmap terrain with multiple levels of detail.
- Store world metadata in structured files alongside terrain data so gameplay systems can later query regions, settlements, roads, and resources.
- Generate terrain into editable chunk assets, then load those chunks through a viewer scene with streaming and LOD.
